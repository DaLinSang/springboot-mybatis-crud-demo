### 4.1 批量插入测试

#### 测试SQL

```java
// 使用MyBatis BATCH执行器
SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH);
OrdersMapper mapper = sqlSession.getMapper(OrdersMapper.class);
for (int i = 1; i <= total; i++) {
    Orders order = generateRandomOrder(startId + i - 1);
    mapper.insert(order);
    if (i % batchSize == 0) {
        sqlSession.flushStatements();
        sqlSession.commit();
    }
}
```

#### 性能分析

### 4.6 多表联查测试

#### 测试SQL

```sql
-- 多表联查1：INNER JOIN 查询有订单的用户及订单数量
SELECT
    u.id as userId,
    u.username,
    u.age,
    COUNT(o.id) as orderCount,
    COALESCE(SUM(o.total_amount), 0) as totalSpent
FROM user_transaction_log u
INNER JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.username, u.age
ORDER BY orderCount DESC
LIMIT #{limit}

-- 多表联查2：LEFT JOIN 查询所有用户（含无订单的）
SELECT
    u.id as userId,
    u.username,
    u.age,
    COUNT(o.id) as orderCount,
    COALESCE(SUM(o.total_amount), 0) as totalSpent
FROM user_transaction_log u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.id BETWEEN #{beginId} AND #{endId}
GROUP BY u.id, u.username, u.age

-- 多表联查3：按年龄段统计订单
SELECT
    u.age,
    COUNT(o.id) as orderCount,
    AVG(o.total_amount) as avgAmount,
    SUM(o.total_amount) as totalAmount
FROM user_transaction_log u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.age
ORDER BY u.age

-- 多表联查4：查询消费总额TOP N用户
SELECT
    u.id as userId,
    u.username,
    u.age,
    COUNT(o.id) as orderCount,
    COALESCE(SUM(o.total_amount), 0) as totalSpent
FROM user_transaction_log u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.username, u.age
ORDER BY totalSpent DESC
LIMIT #{limit}
```

#### 测试结果

| 序号 | 查询类型          | 查询范围            | 耗时(ms) |
| ---- | ----------------- | ------------------- | -------- |
| 1    | INNER JOIN        | 全表，limit=20      | 4682     |
| 2    | LEFT JOIN         | ID范围450000-470000 | 671      |
| 3    | INNER JOIN + 聚合 | 全表，按年龄段分组  | 6119     |
| 4    | LEFT JOIN + 排序  | limit=20            | 18082    |
| 5    | 多表统计汇总      | 全表计数            | 1446     |

#### 性能分析

- **INNER JOIN（4682ms）**：全表扫描50万用户数据并关联订单表，耗时较长
- **LEFT JOIN（671ms）**：范围查询只扫描2000条记录，耗时最短，但耗时时长随着所需记录数增加而增加，为线性增长
- **INNER JOIN + 聚合（6119ms）**：全表联查并按年龄段分组聚合，需要扫描和计算，聚合开销大，耗时长
- **LEFT JOIN + 排序（18082ms）**：联查后进行ORDER BY排序，排序开销巨大，耗时长
- **多表统计汇总（1446ms）**：只做COUNT统计，不需要返回明细数据，性能较好

## 6. 结论与建议

### 6.1 测试结论

1. **批量插入性能稳定**：使用BATCH执行器，插入速度稳定在5000-7000条/秒
2. **分页查询差距巨大**：普通OFFSET分页在深度翻页时性能急剧下降，主键分页和游标分页性能稳定
3. **索引是关键**：有无索引查询性能相差数百倍，尽量为常用查询字段建立索引
4. **多表联查开销显著**：多表联查耗时是单表查询的数倍至数十倍
5. **排序是性能杀手**：带ORDER BY的多表联查耗时最长（18秒），应尽量避免
6. **聚合查询较优**：只做统计的多表查询（1.4秒）比返回明细的联查快很多

### 6.2 性能优化建议

| 问题 | 建议方案 | 预期效果 |
| ---------- | --------------------- | ------------------ |
| 深度分页慢 | 使用游标分页替代LIMIT | 性能提升较大 |
| COUNT(*)慢 | 使用近似值或Redis缓存 | 能够实现毫秒级 |
| 无索引查询慢 | 为常用字段建索引 | 查询提升数百倍 |
| 多表联查慢 | 减少联表数量，只查必要字段 | 视情况优化 |
| 联查+排序极慢 | 避免大数据量多表排序，考虑预计算 | 视情况优化 |

### 7.2 MyBatis BATCH执行器原理

```java
// BATCH执行器工作原理
SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH, false);
OrdersMapper mapper = sqlSession.getMapper(OrdersMapper.class);

// 循环插入，数据先进入缓冲区
for (int i = 1; i <= total; i++) {
    mapper.insert(order);  // 不立即执行，积累在缓冲区
    if (i % batchSize == 0) {
        sqlSession.flushStatements();  // 一次性发送1000条SQL
        sqlSession.commit();           // 提交事务
    }
}
```

**与SIMPLE模式的区别**：
- SIMPLE模式：每条SQL单独发送，50万条=50万次网络IO
- BATCH模式：每1000条打包发送，50万条=500次网络IO
